## What I checked — Batch 8 China-market CLIs
- `codebuddy`: Tencent CodeBuddy CLI memory docs (https://www.codebuddy.ai/docs/cli/memory) confirm recursive `CODEBUDDY.md`/`CODEBUDDY.local.md` from cwd upward plus user-level `~/.codebuddy/CODEBUDDY.md`; `AGENTS.md` is an explicit fallback only when no `CODEBUDDY.md` exists ("If a `CODEBUDDY.md` file exists in the project, project-level memory will use CODEBUDDY.md, otherwise it will use AGENTS.md"). Own-name preferred → `symlink`. Toolkit's `skill_agents.py:114-121` aligns on `.codebuddy/` paths.
- `codestudio`: Confirmed via repeated searches and prior subagent fragment (`docs/agent-toolkit/research/subagent-fragments/batch-8-china-market.md:18`) — no product named "codestudio" exists across Tencent/ByteDance/Alibaba/Baidu/Volcano Engine catalogues. The toolkit entry (`skill_agents.py:129-135`) uses speculative `.codestudio/` paths. Verdict stands at `unknown`.
- `forgecode`: ForgeCode custom-rules guide (https://forgecode.dev/docs/custom-rules-guide/) confirms automatic `AGENTS.md` discovery in three locations (env `base_path` → git root → cwd) with no flags. No separate user/global scope is documented for the CLI — rules live at the resolved project root only. Native AGENTS.md reader.
- `hermes-agent`: Nous Research docs (https://hermes-agent.nousresearch.com/docs/guides/tips) and the cross-check via DeepWiki/Skywork explicitly state "Hermes loads the top-level `AGENTS.md` from the current working directory at session start". Subdirectory `AGENTS.md` files inject lazily via tool results. Global personality is a different file (`~/.hermes/SOUL.md`), NOT `AGENTS.md` at user scope. This OVERTURNS the existing subagent-fragment verdict of `unsupported (by design)` — that fragment was about subagent routing (delegate_task tool), not about instruction-file loading; for the `instructions` kind specifically, Hermes is `native`.
- `kiro-cli`: Kiro Steering docs (https://kiro.dev/docs/cli/steering/) explicitly state `AGENTS.md` files "do not support inclusion modes and are always included" and are picked up from both `~/.kiro/steering/` and the workspace root automatically. Issue #6755 is closed/duplicate and concerns other steering files (`*.md` in `.kiro/steering/`, skill defs) — the docs carve out `AGENTS.md` as the one always-included file. Native.
- `augment` (Auggie CLI): Augment CLI rules doc (https://docs.augmentcode.com/cli/rules) gives an explicit precedence chain — `--rules` flag → `CLAUDE.md` → `AGENTS.md` → `.augment-guidelines` → `.augment/rules/` (project) → `~/.augment/rules/` (user). `CLAUDE.md` is consulted before `AGENTS.md`, so the "first/most-prominent" project-root file is the fixed own-name `CLAUDE.md` (brief rule 5); the v0.16.0 release notes added a "hierarchical rules indicator" for `AGENTS.md` but that doesn't change precedence. User-scope is a directory (`~/.augment/rules/`), not a single root file. Per brief tie-breaker "when in doubt lean symlink", → `symlink` with target `CLAUDE.md`.
