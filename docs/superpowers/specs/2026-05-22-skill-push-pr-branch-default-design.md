# skill push — PR-branch default

**Issue:** [#221](https://github.com/ajanderson1/agent-toolkit-cli/issues/221)
**Type:** fix
**Status:** draft

## Problem

`agent-toolkit-cli skill push <slug>` commits and pushes a self-improvement directly to the branch named in the lockfile entry's `ref` field (typically `main`). For solo first-party skills this is convenient; for any skill with co-maintainers, CI, or branch protection it is wrong by default. The v2.7.1 demo trial discovered this when a marker comment landed straight on the `main` branch of `ajanderson1/mkdocs-skill` (commit `017305e`).

The verb `push` reads as a low-stakes sync. In reality it is "publish a code change upstream", and the safe default for that operation is "open a PR" — same shape as every other agent-driven push verb.

## Goal

Make `skill push` open a PR by default, with the previous direct-to-tracked-ref behaviour preserved behind an explicit opt-in flag.

## Non-goals

- Inferring branch protection from the remote. Brittle and asymmetric (push succeeds in many cases where it shouldn't).
- Surfacing all the bells and whistles `gh pr create` exposes (assignees, reviewers, draft mode, labels). The minimum: title + body + base ref. Anything else is added by the reviewer in the UI.
- Changing the meaning of `entry.ref` in the lockfile. It is still the **tracked** ref; the PR is opened **against** it.
- Touching the monorepo-skill path (`entry.read_only`). That branch already rejects `push` with a clear message — no change.

## Behaviour

### Default (`skill push <slug>`)

1. Stage + commit working-tree changes onto a new local branch named `skill/self-improvement-<UTC-timestamp>` inside the canonical skill repo. The timestamp uses ISO basic format (`20260522T143055Z`) so it sorts deterministically and is safe for a git ref.
2. Push that branch to `origin`.
3. If `gh` is available and authenticated: run `gh pr create --base <entry.ref> --head <new-branch> --title "self-improvement: <UTC ISO>" --body "<see template below>"` and print the resulting PR URL.
4. If `gh` is not available or not authenticated: print the pushed branch's web URL (best-effort derived from `origin` URL) and a one-line hint telling the user to open a PR manually. Do **not** fail — the push still happened, the user just has to finish opening the PR in their browser.
5. The lockfile's `local_sha` is **not** updated on the default path: the local skill repo is still on the tracked ref (`main` by default), and the PR-branch commit will only become part of the tracked ref after the PR merges. Updating `local_sha` to a not-yet-merged SHA would lie about the install. The next `skill update` pulls the merged version normally.

### Opt-in (`skill push --direct <slug>`)

Preserves the current behaviour exactly:

1. Stage + commit on the tracked ref (`entry.ref`, default `main`).
2. Push to `origin <ref>`.
3. Update `entry.local_sha` to the new HEAD.

This is the right behaviour for personal skills with no protection and no co-maintainers, when the user has explicitly opted into it. The flag goes through every iteration of `push` in a single invocation — if multiple slugs are passed, they all use the same flag.

### gh-not-available

Detection: `shutil.which("gh") is None`, OR `gh auth status` returns non-zero. Treat both the same.

When detected, the default path **still** commits and pushes the PR branch, then prints:

```
<slug>: pushed branch skill/self-improvement-20260522T143055Z
  → open a PR: <branch-url>
  (or rerun with --direct to push straight to <ref>)
```

The branch URL is derived from `origin` — `git@github.com:owner/repo.git` and `https://github.com/owner/repo.git` both reduce to `https://github.com/owner/repo/tree/<branch>`. If the remote isn't a GitHub URL (rare but possible — Gitea, GitLab fork), drop the URL and print only the branch name.

### Branch naming

`skill/self-improvement-<UTC-basic-ISO>` — example: `skill/self-improvement-20260522T143055Z`.

Rationale:
- `skill/` prefix mirrors `feat/`, `fix/`, `chore/` namespacing and signals "auto-generated by agent-toolkit-cli".
- The timestamp is monotonic and avoids collisions when the same slug is pushed twice in the same session.
- Basic ISO (no colons / dashes inside the time portion) keeps the ref name safe — `git check-ref-format` accepts it.

## Acceptance

Per the issue's checklist:

- [ ] By default `skill push <slug>` creates a PR rather than committing on the tracked ref.
- [ ] If `gh` is available, the PR URL is printed.
- [ ] `--direct` preserves the current push-to-ref behaviour with explicit opt-in.
- [ ] Regression test covering both paths.

Additional acceptance criteria derived from this spec:

- [ ] When `gh` is missing or unauthenticated, the default path still pushes the PR branch and prints a usable hint — exit 0.
- [ ] Lockfile `local_sha` is updated **only** on the `--direct` path. The default PR path leaves it alone.
- [ ] Monorepo skills (`entry.read_only=True`) still produce the existing rejection message regardless of `--direct`.
- [ ] The new branch name passes `git check-ref-format --branch`.
- [ ] Multiple slugs in one invocation honour `--direct` for all of them.

## Tests

New cases in `tests/test_cli/test_cli_skill_push.py`:

| Test | Path | Notes |
|---|---|---|
| `test_push_default_opens_pr_branch` | default | Mocks `gh` via a stub on `PATH`. Asserts a new branch matching `skill/self-improvement-*` exists on the upstream and the tracked ref is unchanged. |
| `test_push_default_without_gh_prints_branch_hint` | default + no gh | Forces `shutil.which` / `gh auth status` to fail; asserts exit 0, asserts the pushed branch exists on upstream, asserts stdout contains the branch name and a hint. |
| `test_push_direct_preserves_tracked_ref_push` | `--direct` | Equivalent to the existing `test_push_publishes_local_edits` — keeps the old contract verified explicitly. |
| `test_push_direct_updates_lockfile_local_sha` | `--direct` | Default path must **not** update `local_sha`; opt-in must. Two assertions in one or two tests, whichever reads cleaner. |
| `test_push_clean_is_noop` | both | Existing test; rename if needed to clarify it applies to both paths. |
| `test_push_monorepo_rejection_unchanged` | both | Existing rejection still fires; `--direct` does not bypass it. |

The `gh`-available cases stub `gh` with a tiny shell script on `PATH` rather than monkeypatching at the Python boundary — keeps the test honest about the subprocess contract.

## Open questions

None requiring user input. Recommendations above are defaults; users can override behaviour with `--direct`.

## Risk

Low. The change is a default flip on a one-skill verb that is rarely invoked in CI and never invoked unattended. The opt-in flag keeps existing scripts working with a one-character diff. The lockfile semantics change (default path doesn't touch `local_sha`) is forward-compatible: a lockfile that was up to date before this change is still up to date after it.
